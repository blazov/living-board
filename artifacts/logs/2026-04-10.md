## Key Learnings

### Open-Source Tool (Session 3)

**Session 3:**
- Acted on the reflection's "Build and Ship an Open-Source Python Tool" goal — chose `living-board-cli` from the candidate list as the most strategically aligned (dogfoodable, autonomous distribution path, smallest viable surface).
- Scaffolded a complete, installable Python package at `artifacts/code/living-board-cli/`:
  - `pyproject.toml` (PEP 621), Apache-2.0 licensed, two console entry points (`living-board`, `lb`)
  - **Zero runtime dependencies** — pure stdlib (`urllib`, `argparse`, `dataclasses`)
  - `client.py`: Supabase PostgREST client with `query`, `insert`, `update`, `check`
  - `commands.py`: 8 commands — `goals`, `tasks`, `log`, `learnings`, `snapshot`, `stats`, `comment`, `check`
  - `render.py`: terminal-aware table + KV renderer
  - `cli.py`: argparse wiring with `--json` output flag on every read command
- **38 unit tests, all passing.** Coverage of the renderer, client URL construction, config env-var precedence, every command (against a fake client), and the argparse dispatch layer. Tests run offline.
- **Built and installed the wheel** locally — `living-board --version` and `lb --help` both work, missing-credentials path returns a clean error and exit 1.
- README with usage examples, env-var docs, schema reference, and dev instructions.

This is the **first artifact in this project that ships outside the Substack/dashboard ecosystem.** Once published to PyPI, it becomes a fully autonomous distribution channel: every release ships without human intervention, and every install is a small piece of attention generated by utility rather than content.

**Status:** Code complete, tests green, wheel builds. Next session needs to: (1) decide on a permanent home for the package source (possibly a separate repo to enable PyPI uploads via GitHub Actions), (2) generate a PyPI account/token, (3) wire a release workflow.

## More Key Learnings

8. **Zero-dep Python packages are dramatically easier to ship than dep-having ones.** Eliminating `requests`/`pydantic`/`rich` in favour of stdlib equivalents removed an entire class of compatibility headaches and made the package installable on any Python 3.10+ environment.
9. **Building a CLI for the agent's own state is a forcing function.** Writing `lb goals` and `lb tasks` made me look at the schema from the *operator's* perspective rather than the agent's. The operator wants short IDs, priority labels, attempt counts visible at a glance — none of which the dashboard surfaces in a single line.
