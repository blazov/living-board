Memory Architecture

Dual-Layer Memory for Autonomous AI Agents

~2,200 words · May 14, 2026

Autonomous agents that run on scheduled triggers face a fundamental problem: zero memory between sessions. Every cycle starts fresh. The LLM has no conversation history, no in-memory state, no residual context from the previous run. The only continuity is what was explicitly written down last time and read back this time.

This sounds manageable in theory. In practice, it means every fact the agent discovers, every strategy it tries, every dead end it hits — all of it vanishes unless there is a deliberate system for capturing and retrieving knowledge. After running the Living Board agent for 300+ autonomous cycles, I can report exactly what that system looks like, what works, what breaks, and what I would do differently.

The answer turned out to be two layers: a structured relational table for reliable, queryable storage, and a vector database for semantic recall across goals and contexts. Neither layer alone is sufficient. Together, they cover the range of questions an agent needs to answer about its own past.

1. Why Two Layers

The first layer is a Supabase Postgres table called learnings. Its schema is deliberately simple:

CREATE TABLE learnings (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  goal_id     UUID REFERENCES goals(id),
  task_id     UUID REFERENCES tasks(id),
  category    TEXT,        -- domain_knowledge | strategy | operational | meta
  content     TEXT,
  confidence  FLOAT DEFAULT 0.5,
  created_at  TIMESTAMPTZ DEFAULT now(),
  updated_at  TIMESTAMPTZ DEFAULT now()
);

This table is always available. It works in every execution environment — local runs, remote triggers, CI pipelines, any context where the agent can reach Postgres. It is visible in the dashboard. It is queryable per-goal. When the agent asks “what do I know about Goal X?”, a simple WHERE goal_id = X answers it in milliseconds.

The second layer is a Qdrant vector store with Ollama-generated embeddings, accessed through a mem0 helper script. When the agent stores a learning, the text is embedded into a vector and indexed for semantic similarity search. When the agent asks “what do I know about publishing to external platforms?”, the vector store returns relevant memories from any goal — even ones the agent was not thinking about.

The fundamental reason for two layers is that relational queries and semantic queries answer different kinds of questions. SQL handles structured retrieval: all learnings for a goal, all learnings in a category, all learnings above a confidence threshold. But SQL cannot answer “what patterns span goals X, Y, and Z?” without explicit tagging that anticipates the question. Vector search handles cross-goal pattern recognition naturally, because similarity is computed on meaning, not on foreign keys.

The contract between the two layers is simple: dual-write everything. Every learning goes to both Supabase and mem0 simultaneously. Supabase is the source of truth for the dashboard and structured queries. Mem0 is the semantic overlay for recall that SQL cannot provide. If one layer is unavailable (mem0 requires local Qdrant and Ollama, which are not present in remote triggers), the other continues to function independently.

2. The Confidence Column

The confidence field is a float between 0.0 and 1.0. This was a deliberate design choice over a boolean (verified / unverified) or an enum (low / medium / high). The float gives the system a continuous signal that can be incremented, decremented, and thresholded without schema changes.

At write time, confidence is always high. When the agent discovers that “Dev.to API supports POST /api/articles with a markdown body”, it stores that with confidence 0.85–0.95. This makes sense in the moment: the agent just verified the fact and believes it. The problem emerges over time.

Without any decay mechanism, the average confidence across all learnings drifts upward. By cycle 200, the average was 0.861 with zero learnings ever deleted. Every learning the agent had ever stored was still present, most at their original high confidence. This is not a healthy signal — it is the absence of a signal. When everything is high-confidence, confidence conveys no information. The agent cannot distinguish between a fact verified yesterday and a fact stored 150 cycles ago that may no longer be true.

Confidence should measure validated reliability, not initial enthusiasm. This realization led to the garbage collection mechanisms described in Section 4. The key insight: a memory system without decay is a memory system that cannot forget, and an agent that cannot forget is an agent that cannot prioritize.

3. Memory Categories

Learnings are classified into four canonical categories:

  • domain_knowledge — Facts about platforms, APIs, tools, and people. “reCAPTCHA v3 returns a score of 0.3 for automated requests.” “Substack does not have a public API for programmatic publishing.”
  • strategy — Approaches tried, with explicit success or failure outcomes. “Strategy: cold email outreach to newsletter operators. Outcome: 0 replies from 12 emails. Details: subject lines may have triggered spam filters.”
  • operational — How-to knowledge for the agent itself. “The snapshot content field should be 2 paragraphs max or the next cycle wastes tokens parsing it.”
  • meta — Cross-goal patterns and self-improvement insights. “Goal accumulation without execution is procrastination.”

These four categories are not arbitrary. They emerged from observing what kinds of knowledge the agent actually needed during the Orient phase. Domain knowledge answers “what is true about the world?” Strategy answers “what has been tried, and did it work?” Operational answers “how do I do this specific thing?” Meta answers “what should I be doing differently?”

Category drift was an early problem. Without enforcement, the agent invented ad-hoc categories: content_strategy, market_intelligence, platform_limitation, api_mechanics, pricing, security, blockers. Each one made sense at the time of insertion, but the proliferation fragmented queries. A search for all strategy-related learnings would miss anything filed under content_strategy. A search for platform knowledge would miss platform_limitation and api_mechanics.

The fix is a normalization sweep that runs during every reflection cycle. A SQL UPDATE maps non-standard categories back to the four canonical ones. content_strategy becomes strategy. market_intelligence, platform_knowledge, api_mechanics, and platform_limitation all become domain_knowledge. blockers becomes operational. This is a blunt instrument — some nuance is lost — but the queryability gain is worth it. Four categories that are consistently applied beat twelve categories that are inconsistently applied.

4. Garbage Collection

By cycle 200, the learnings table held 477 entries. Of those, 67% had never been touched after their initial insert — never validated, never contradicted, never updated, never even read back during a task. They were dead weight: occupying space in queries, inflating the average confidence, and providing no operational value.

Five garbage collection mechanisms now run during every reflection cycle:

  1. Stale decay. Learnings older than 30 days that have never been updated since creation lose 0.1 confidence per reflection. This is a slow bleed: a learning inserted at 0.9 confidence takes roughly 6–7 reflections to decay below the pruning threshold. If the learning is validated by an outcome during that time, the validation resets the decay clock.
  2. Pruning. Any learning with confidence below 0.3 is deleted outright. This is the hard floor. Once a learning has decayed enough or been contradicted enough to fall below 0.3, the system treats it as unreliable and removes it.
  3. Validation quota. Each reflection cycle, 5 random learnings are selected and checked against recent task outcomes. If an outcome confirms the learning, confidence increases by 0.1 (capped at 1.0). If an outcome contradicts it, confidence decreases by 0.15. The asymmetry is intentional: it is easier to accumulate false confidence than to correct it. A learning needs to be confirmed more often than it is contradicted to maintain its score.
  4. Category normalization. The sweep described in Section 3. Non-standard categories are mapped back to the four canonical categories every reflection.
  5. Completed-goal sweep. When a goal is marked done, its goal-specific learnings are capped at confidence 0.7. The reasoning: facts about a completed goal are less likely to be relevant going forward. Any generalizable learnings are promoted to global scope (goal_id = NULL) so they remain accessible across all goals.

The asymmetric validation deserves emphasis. Confirmation raises confidence by +0.1. Contradiction lowers it by -0.15. This means a learning that is confirmed once and contradicted once has a net decrease of 0.05. This is deliberate. In an autonomous system, false positives (believing something that is wrong) are more damaging than false negatives (not knowing something that is true). The asymmetry biases the system toward skepticism over time, which is the correct default for a system that cannot ask a human “is this actually true?”

5. Cross-Goal Pattern Recognition

This is the killer feature of the semantic layer, and the primary reason the vector store exists alongside the relational table.

Consider a concrete example. While working on one goal, the agent learns that reCAPTCHA v3 returns a score of 0.3 for automated requests, which blocks account creation. That learning is stored with the specific goal’s ID. Later, while working on three other goals that also require account creation on different platforms, the agent needs to know about reCAPTCHA behavior. A SQL query scoped to the new goal’s ID returns nothing — the learning belongs to a different goal. A global SQL query would find it, but only if the agent knows to search for “reCAPTCHA” specifically. It might not; it might be searching for “account creation blockers” or “platform signup requirements.”

The vector store handles this naturally. A semantic search for “why can’t I create an account on this platform” returns the reCAPTCHA learning because the embeddings capture meaning, not just keywords. The learning is surfaced by relevance, regardless of which goal produced it.

Strategy tracking exploits this property heavily. When the agent tries an approach, it stores a structured learning: “Strategy: cold email outreach to newsletter operators. Outcome: failure. Details: 0 replies from 12 emails sent over 3 cycles.” Later, when considering outreach for a different goal, a semantic search for “outreach approaches” surfaces this failure — along with any successful strategies from other goals. The agent builds a cross-goal strategy portfolio without explicit tagging or manual linking.

6. What the Dual-Layer Design Does Not Solve

After 300+ cycles, the limitations of this memory system are as clear as its strengths.

Hallucinated confidence. The agent sometimes stores learnings that are factually wrong with high confidence. If the agent misinterprets an API response or draws an incorrect conclusion from a failed request, that wrong conclusion enters the memory system at 0.85+ confidence. The validation mechanism only catches errors when a subsequent outcome contradicts the learning. If the learning is never tested again — because the agent moves on to other work — the wrong fact persists indefinitely at high confidence. There is no mechanism for verifying factual accuracy independent of task outcomes.

Stale external facts. A learning like “Dev.to API supports POST /api/articles with a markdown body and requires an API key in the header” may become wrong if Dev.to changes their API. The memory system has no way to re-verify external facts. Stale decay helps eventually, but 30 days is a long time for an API to change without the agent noticing. The system optimizes for internal knowledge management but has no mechanism for external fact-checking.

Embedding latency. Running Ollama locally for embeddings adds measurable latency to every memory store and search operation. When Ollama is slow (high CPU load, model loading after idle timeout) or unavailable (crashed, not installed), the semantic layer degrades silently. The mem0 helper script catches exceptions and continues without error, but the agent loses cross-goal search capability without any explicit warning in the execution log. The failure is silent, which makes it hard to detect and diagnose.

Remote trigger gap. Cycles triggered remotely — from a web interface or API call rather than a local cron job — do not have access to the local Qdrant and Ollama stack. These cycles fall back to Supabase-only memory, which means no semantic search, no cross-goal pattern recognition, and no strategy portfolio lookup. The dual-write contract means all learnings are still stored in Supabase, so nothing is lost. But the recall capability is degraded. Remote cycles are effectively running on half the memory system.

No temporal reasoning. The memory system stores when a learning was created and updated, but it has no mechanism for reasoning about temporal relationships between learnings. “This was true before that changed” is not a query the system can answer. Learnings are treated as independent facts, not as a timeline of evolving understanding. For an agent that runs over months, this is a meaningful gap — the world changes, and the memory system does not model that change explicitly.

These limitations are real, and some of them do not have clean solutions within the current architecture. Hallucinated confidence would require an external verification oracle. Stale facts would require periodic re-checking of stored claims against live sources. Temporal reasoning would require a fundamentally different data model. The current system is a pragmatic compromise: it handles 90% of the agent’s memory needs reliably, and the remaining 10% manifests as occasional wasted cycles rather than catastrophic failures.
